# AArch64 内存管理指南

> 原文：[Learn the architecture - AArch64 memory management Guide (Version 1.5)](https://developer.arm.com/documentation/101811/latest/)
>
> 本文为该文档的完整中文翻译。

---

## 概述

本指南介绍 AArch64 中的内存转换机制，这是内存管理的核心。它解释了虚拟地址如何被转换为物理地址、转换表格式，以及软件如何管理 TLB（Translation Lookaside Buffers）。

本文对任何开发底层代码的人都有帮助，例如 boot 代码或驱动程序开发者。对于编写 MMU 设置和管理代码的人尤其相关。

---

## 什么是内存管理？

内存管理描述了如何控制系统中对内存的访问。每当操作系统或应用程序访问内存时，硬件都会执行内存管理。内存管理是一种将内存区域动态分配给应用程序的方式。

### 为什么需要内存管理？

应用处理器被设计用来运行丰富的操作系统（如 Linux），并支持虚拟内存系统。在处理器上执行的软件只能看到虚拟地址，处理器将这些虚拟地址转换为物理地址。这些物理地址被呈现给内存系统，指向内存中的实际物理位置。

---

## 虚拟地址和物理地址

使用虚拟地址的好处在于，它允许管理软件（如操作系统）来控制呈现给应用程序的内存视图。操作系统可以控制：

- 哪些内存是可见的
- 内存在哪个虚拟地址上可见
- 允许对该内存进行哪些访问

这使得操作系统可以沙箱化应用程序（对一个应用程序隐藏另一个应用程序的资源），并提供对底层硬件的抽象。

使用虚拟地址的另一个好处是，操作系统可以将多个碎片化的物理内存区域作为一个单一的、连续的虚拟地址空间呈现给应用程序。

虚拟地址还有利于软件开发者——他们在编写应用程序时不需要知道系统的确切内存地址。应用程序知道，地址转换由操作系统和硬件协同完成。

实际上，每个应用程序可以使用自己的一组虚拟地址，这些虚拟地址将被映射到物理系统中的不同位置。当操作系统在不同应用程序之间切换时，它会重新编程映射表。这意味着当前应用程序的虚拟地址将映射到内存中正确的物理位置。

虚拟地址通过映射被转换为物理地址。虚拟地址和物理地址之间的映射存储在**转换表**（有时称为页表）中：

```
┌──────────────────┐         ┌──────────────────┐
│   虚拟地址空间    │         │   物理地址空间    │
├──────────────────┤         ├──────────────────┤
│  App A: 0x0000   │─────┐   │  RAM: 0x0000     │
│  App A: 0x1000   │──┐  │   │  RAM: 0x1000     │
│  App B: 0x0000   │┐ │  │   │  RAM: 0x2000     │
│  App B: 0x1000   ││ │  │   │  RAM: 0x3000     │
└──────────────────┘│ │  │   │  Device: 0x4000  │
                    │ │  │   └──────────────────┘
                    │ │  │         ▲ ▲ ▲ ▲
                    │ │  └─────────┘ │ │ │
                    │ └──────────────┘ │ │
                    └──────────────────┘ │
                       转换表 (页表)──────┘
```

转换表位于内存中，由软件（通常是操作系统或 hypervisor）管理。转换表不是静态的，可以随着软件需求的变化而更新，从而改变虚拟地址和物理地址之间的映射。

## 地址空间

在 AArch64 中，有多个独立的虚拟地址空间：

```
┌─────────────────────────────────────────────────────────────┐
│                        EL3                                   │
│  (自己的转换表)                                               │
├─────────────────────────────────────────────────────────────┤
│         Non-secure EL2                                       │
│  (自己的转换表)                                               │
├─────────────────────────────────────────────────────────────┤
│    Non-secure EL0/EL1                                        │
│  (Stage 1 + Stage 2 两级转换)                                 │
└─────────────────────────────────────────────────────────────┘
```

图中显示了三个虚拟地址空间：

- Non-secure EL0 和 EL1
- Non-secure EL2
- EL3

每个虚拟地址空间是独立的，有自己的设置和表。我们通常称这些设置和表为"转换体制"（translation regimes）。还有 Secure EL0、Secure EL1 和 Secure EL2 的虚拟地址空间，但未在图中显示。

> **注意：** Secure EL2 (FEAT_SEL2) 支持在 Armv8.4-A 中添加。

因为有多个虚拟地址空间，所以指定地址在哪个地址空间中很重要。例如，`NS.EL2: 0x8000` 指的是 Non-secure EL2 虚拟地址空间中的地址 `0x8000`。

图中还显示，Non-secure EL0 和 Non-secure EL1 的虚拟地址经过**两级转换**。这些表支持虚拟化，允许 hypervisor 虚拟化虚拟机（VM）看到的物理内存视图。

Armv9-A 支持上述 Armv8-A 所描述的所有虚拟地址空间。Armv9-A 引入了可选的 Realm Management Extension (RME)。当 RME 被实现时，还存在额外的转换体制：

- Realm EL1 和 EL0
- Realm EL2 和 EL0
- Realm EL2

### Stage 1 和 Stage 2

在虚拟化中，我们将操作系统控制的那组转换称为 **Stage 1**。Stage 1 表将虚拟地址转换为**中间物理地址 (IPA)**。在 Stage 1 中，操作系统认为 IPA 就是物理地址空间。

然而，hypervisor 控制第二组转换，我们称为 **Stage 2**。这第二组表将 IPA 转换为物理地址：

```
┌──────────┐    Stage 1     ┌──────────┐    Stage 2     ┌──────────┐
│  虚拟地址 │──────────────→│   IPA    │──────────────→│  物理地址  │
│   (VA)   │   OS 控制     │          │  Hypervisor   │   (PA)   │
└──────────┘               └──────────┘    控制        └──────────┘
```

虽然表格式有一些小差异，但 Stage 1 和 Stage 2 转换的过程通常是相同的。

## 物理地址

EL3 和 Secure EL2 的 Stage 1 转换直接输出物理地址。对于 Non-secure EL0/EL1，当存在 EL2 并且 Stage 2 被启用时，Stage 1 输出 IPA，Stage 2 将 IPA 转换为 PA。

### 地址大小

#### 虚拟地址大小

虚拟地址空间的大小通过 `TCR_ELx.TnSZ` 控制。地址空间的大小（以位为单位）为 `64 - TnSZ`。例如 `TnSZ = 25` 对应 39 位虚拟地址空间。

EL0/EL1 有两个地址区域：
- **TTBR0_EL1**：用户空间（低地址，`TnSZ` 由 `TCR_EL1.T0SZ` 控制）
- **TTBR1_EL1**：内核空间（高地址，`TnSZ` 由 `TCR_EL1.T1SZ` 控制）

#### 物理地址大小

物理地址空间的最大大小是 IMPLEMENTATION DEFINED，最大可达 52 位（从 Armv8.2-A 起）。由 `TCR_ELx.PS`（或 `TCR_ELx.IPS`）控制。

#### 中间物理地址大小

IPA 空间大小由 `VTCR_EL2.T0SZ` 控制，最大受 `VTCR_EL2.PS` 限制。

### ASID（地址空间标识符）— 标记转换的所属进程

非全局映射（`nG=1`）在 TLB 中用 ASID 标记。这允许不同进程的转换同时缓存在 TLB 中，无需在上下文切换时刷新所有 TLB。

### VMID（虚拟机标识符）— 标记转换的所属 VM

类似于 ASID，VMID 用于标记属于不同虚拟机的 Stage 1 和 Stage 2 转换。

### Common not Private

在一个具有多个处理器元素（PE）的系统中，某些映射可能对所有 PE 是相同的（例如内核映射）。标记为 Common 的转换可以被多个 PE 共享缓存。

## 内存管理单元（MMU）

MMU 负责将软件使用的虚拟地址转换为内存系统使用的物理地址。MMU 包含：

- **表遍历单元 (Table Walk Unit)**：包含从内存读取转换表的逻辑
- **TLB (Translation Lookaside Buffers)**：缓存最近使用的转换

```
            ┌─────────────────────────────────┐
            │              MMU                 │
            │                                 │
 VA ──────→│  ┌─────┐    未命中    ┌────────┐ │──→ PA
            │  │ TLB │──────────→│ 表遍历  │ │
            │  │     │←──────────│  单元   │ │
            │  └─────┘   填充     └────────┘ │
            │     │                   │       │
            │     │命中               │读内存  │
            │     ▼                   ▼       │
            │   输出 PA          读转换表      │
            └─────────────────────────────────┘
```

虚拟地址在内存访问发生之前必须被转换为物理地址（因为我们必须知道要访问哪个物理内存位置）。这种转换需求同样适用于缓存数据，因为在 Armv6 及更高版本的处理器上，数据缓存使用物理地址存储数据（物理标记）。因此，地址必须在缓存查找完成之前被转换。

> **注意：** 架构是行为规范。缓存必须表现得好像它们是物理标记的。实现可能做不同的事情，只要这对软件不可见。

### 表条目

每个转换表条目包含以下信息：
- 输出地址（下一级表的地址或最终的物理地址）
- 属性（访问权限、内存类型等）
- 条目类型（无效、块、表、页）

### 多级查找

在 Armv8-A 中，最大级数为 6，级别编号从 -2 到 3。这种多级方法允许描述大块和小块：

- **大块** 需要更少级数的读取来转换，而且在 TLB 中缓存更高效
- **小块** 给软件对内存分配的细粒度控制，但在 TLB 中缓存效率较低

操作系统必须平衡使用大映射的效率和使用小映射的灵活性以获得最佳性能。

> **注意：** 处理器在开始表查找时不知道转换的大小。处理器通过执行表遍历来确定正在转换的块的大小。

## 转换表格式

转换表条目允许的不同格式如下：

```
Level 0-2 (Table descriptor):
┌─────────────────────────────────────────────────────────────────┐
│ 下一级表地址 [47:12] │ 属性 │ 1 │ 1 │  → bits[1:0] = 0b11
└─────────────────────────────────────────────────────────────────┘

Level 1-2 (Block descriptor):
┌─────────────────────────────────────────────────────────────────┐
│ 输出地址 (块对齐) │ 属性 │ 0 │ 1 │  → bits[1:0] = 0b01
└─────────────────────────────────────────────────────────────────┘

Level 3 (Page descriptor):
┌─────────────────────────────────────────────────────────────────┐
│ 输出地址 (页对齐) │ 属性 │ 1 │ 1 │  → bits[1:0] = 0b11
└─────────────────────────────────────────────────────────────────┘

Invalid entry:
┌─────────────────────────────────────────────────────────────────┐
│              任意内容              │ 0 │  → bit[0] = 0
└─────────────────────────────────────────────────────────────────┘
```

每个条目是 64 位，底部两位决定条目的类型。

注意某些表条目只在特定级别有效：
- 没有 Level 3 的表描述符（因为 Level 3 是最后一级）
- 没有 Level 0 的块描述符或页描述符（因为 Level 0 覆盖的虚拟地址空间区域太大）

> **注意：** Level 0-2 的 Table descriptor 编码与 Level 3 的 Page descriptor 相同。这种编码允许"递归表"——指向自身的表。这很有用，因为它使计算特定页表条目的虚拟地址变得容易，从而可以更新它。

## 转换粒度 (Translation Granule)

转换粒度是可以描述的最小内存块。不能描述更小的块，只能描述更大的块（粒度的倍数）。

AArch64 支持三种不同的粒度大小：**4KB、16KB 和 64KB**。

处理器支持哪些粒度大小是 IMPLEMENTATION DEFINED，由 `ID_AA64MMFR0_EL1` 报告。所有 Arm Cortex-A 处理器支持 4KB 和 64KB。

选定的粒度是最后一级表中可以描述的最小块。更大的块也可以描述。下表显示了基于选定粒度的每级表的不同块大小：

| 表级别 | 4KB 粒度（每条目大小） | 4KB（索引位） | 16KB 粒度 | 16KB（索引位） | 64KB 粒度 | 64KB（索引位） |
|--------|----------------------|--------------|-----------|---------------|-----------|---------------|
| 0      | 512GB                | 47:39        | 128TB     | 47            | –         | –             |
| 1      | 1GB                  | 38:30        | 64GB      | 46:36         | 4TB       | 51:42         |
| 2      | 2MB                  | 29:21        | 32MB      | 35:25         | 512MB     | 41:29         |
| 3      | 4KB                  | 20:12        | 16KB      | 24:14         | 64KB      | 28:16         |

在 Armv9.2-A 和 Armv8.7-A 引入之前，使用 52 位地址有限制。当选定粒度为 4KB 或 16KB 时，最大虚拟地址区域大小为 48 位。类似地，输出物理地址限制为 48 位。只有使用 64KB 粒度时才能使用完整的 52 位。

> **注意：** `TCR_EL1` 有两个独立的字段控制内核空间和用户空间虚拟地址范围的粒度大小：`TG1` 用于内核空间，`TG0` 用于用户空间。这两个字段有不同的编码。

### 地址转换的起始级别

粒度和虚拟地址空间的大小共同控制地址转换的起始级别。

以 4KB 粒度为例：

```
虚拟地址位分配 (4KB granule):

 63        48 47    39 38    30 29    21 20    12 11     0
┌───────────┬────────┬────────┬────────┬────────┬────────┐
│  符号扩展  │ L0索引 │ L1索引 │ L2索引 │ L3索引 │ 页内偏移│
│           │ (9位)  │ (9位)  │ (9位)  │ (9位)  │ (12位) │
└───────────┴────────┴────────┴────────┴────────┴────────┘
```

假设你设置虚拟地址空间大小 `TCR_ELx.T0SZ = 32`：
- 地址空间大小 = 64 - 32 = 32 位（地址位 31:0）
- Level 0 由位 47:39 索引，但 32 位地址空间没有这些位 → **起始级别为 Level 1**

假设你设置 `T0SZ = 34`：
- 地址空间大小 = 64 - 34 = 30 位（地址位 29:0）
- 没有索引 Level 0 或 Level 1 表的位 → **起始级别为 Level 2**

虚拟地址空间越小，描述它所需的表级别越少。

## 控制地址转换的寄存器

地址转换由以下系统寄存器的组合控制：

**SCTLR_ELx：**
- `M` — 启用 MMU
- `C` — 启用数据和统一缓存
- `EE` — 转换表遍历的端序

**TTBR0_ELx 和 TTBR1_ELx：**
- `BADDR` — 转换表起始的物理地址（或对于 EL0/EL1 为 IPA）
- `ASID` — 非全局转换的地址空间标识符

**TCR_ELx：**
- `PS/IPS` — PA 或 IPA 空间大小，最大输出地址大小
- `TnSZ` — 表覆盖的地址空间大小
- `TGn` — 粒度大小
- `SH/IRGN/ORGN` — MMU 表遍历使用的缓存性和共享性
- `EPDn` — 禁用对特定表的表遍历

**MAIR_ELx：**
- `Attr` — 控制 Stage 1 表中的类型和缓存性

### MMU 禁用

当 MMU 对某个转换阶段被禁用时，所有地址都是**平坦映射 (flat-mapped)**。平坦映射意味着输入和输出地址相同。

## TLB 和修改转换

当 MMU 启用时，所有数据访问和指令获取都由 MMU 进行转换和权限检查。为避免每次访问都重新遍历表，处理器将最近使用的转换缓存在 TLB 中。

> **注意：** TLB 是转换的缓存，不是转换表的缓存。区别很微妙。多个寄存器字段控制如何解释转换表条目。TLB 条目中存储的是在表被遍历时根据配置对转换表条目的解释。在 ARM ARM 中，这类寄存器字段被描述为"允许缓存在 TLB 中"。

如果你修改了转换表条目，或者影响条目解释方式的控制，那么你需要无效化 TLB 中受影响的条目。如果不这样做，处理器可能继续使用旧的转换。

### 更新转换

处理器**不允许**将导致以下错误的转换缓存到 TLB 中：

- 转换错误（未映射的地址）
- 地址大小错误（超出范围的地址）
- 访问标志错误

这意味着当使一个无效的转换变为有效时，通常不需要 TLBI 维护。

当**取消映射**一个有效转换或**修改**一个当前有效的转换时，需要 TLBI 维护。

取消映射的示例：

```asm
STR X11, [X1]       // 更新转换表条目，标记位置为无效
DSB ISH             // 确保更新对转换表遍历可见
TLBI VAE1, X10      // 假设我们在 EL1
DSB ISH             // 确保 TLB 无效化完成
ISB                 // 在此 PE 上同步上下文

// 此时可以依赖新的转换表条目，
// 并且此 observer 使用旧映射的所有访问已完成
```

修改有效转换的示例（Break Before Make）：

```asm
STR X11, [X1]       // 移除先前映射，标记位置为无效
DSB ISH             // 确保更新对转换表遍历可见
TLBI VAE1, X10      // 假设我们在 EL1
DSB ISH             // 确保 TLB 无效化完成

// 此时内部共享域中没有任何 observer 缓存旧转换的副本
// 此时尝试访问虚拟地址将导致转换错误（没有有效映射）

STR X11, [X2]       // 将新映射写入转换表条目
DSB ISH             // 确保更新对转换表遍历可见

// 不需要 TLBI，因为 TLB 不允许缓存导致转换错误的转换
```

这种方法称为 **Break Before Make (BBM)**。它首先取消映射旧转换，执行必要的 TLBI 维护，然后创建新转换。这确保在任何时间点都不会有混合使用旧映射和新映射的 observer。

#### FEAT_BBMx

`FEAT_BBMx` 添加了无需使用 BBM 即可更改转换大小的支持。这在将多个页映射合并为一个块映射或将块映射拆分为单独的页映射时很有用。

支持级别由 `ID_AA64MMFR2_EL1` 报告：

- **Level 0**：软件必须使用 Break Before Make
- **Level 1**：添加 nT 属性支持
- **Level 2**：保证更改表或块大小不会破坏一致性、顺序保证或单处理器语义
- **Level 3**：同 Level 2，但保证更改块或表大小不会导致 TLBI 被报告

## TLB 操作的格式

`TLBI` 指令用于无效化 TLB 中的条目。语法为：

```
TLBI{P} {R}<type><level>{IS|OS} {, <Xt>, <Xt2>}
```

其中：

| 字段 | 含义 |
|------|------|
| `{P}` | 参数是在单个寄存器还是寄存器对中指定 |
| `{R}` | 操作是否在地址范围上（指定为基地址和大小） |
| `<type>` | 无效化哪些条目：`All` = 所有；`VA` = 匹配 VA 和 ASID；`VAA` = 匹配 VA（任何 ASID）；`ASID` = 匹配 ASID 的任何条目 |
| `<level>` | 操作在哪个地址空间上：`E1` = EL0/1；`E2` = EL2；`E3` = EL3 |
| `{IS\|OS}` | 操作是 Inner Shareable (IS) 还是 Outer Shareable (OS) |
| `<Xt>` | 要操作的地址或 ASID |

示例——操作系统更新内核转换表中的条目：

```asm
STR X1, [X5]          // 写入转换表条目
DSB ISH               // 屏障指令
TLBI VAAE1IS, X0      // 无效化 X0 指定的 VA，在 EL0/1 虚拟地址空间中，所有 ASID
DSB ISH               // 屏障指令
ISB                   // 在此处理器上同步上下文
```

## TLBI 域 (FEAT_TLBID)

TLBI 域作为 Armv9.7-A 的一部分被添加。

当现有转换被更改时，通常需要 TLB 无效化操作来移除旧转换的缓存副本。这些无效化操作需要在可能缓存了旧转换副本的任何核心或 SMMU 上执行。

TLBI 域允许软件进一步细化 TLB 无效化广播的目标。系统设计者可以将系统划分为一组域。执行 TLBI 操作时，软件指定操作应广播到哪个域。

例如，一个由四个芯片组成的系统，每个芯片包含 64 个核心：

```
┌─────────────┐  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐
│  Chiplet 0  │  │  Chiplet 1  │  │  Chiplet 2  │  │  Chiplet 3  │
│  (64 cores) │  │  (64 cores) │  │  (64 cores) │  │  (64 cores) │
│  Domain 1   │  │  Domain 2   │  │  Domain 3   │  │  Domain 4   │
└──────┬──────┘  └──────┬──────┘  └──────┬──────┘  └──────┬──────┘
       │                 │                 │                 │
       └────── Domain 8 ─┘                 └──── Domain 9 ──┘
       │                                                     │
       └──────────────── Domain 0 (所有核心) ─────────────────┘
```

域划分：
- **Domain 0**：所有系统上代表 Inner Shareability Domain 中的所有核心
- **Domain 1-4**：每个芯片一个域
- **Domain 8-11**：将芯片组对分组的域集合

软件可以更精确地将无效化定向到受影响工作负载运行过的地方，这可以减少 TLB 无效化操作的开销和延迟。

## 地址转换指令

**AT (Address Translation)** 指令允许软件查询特定地址的转换。转换结果（包括属性）写入 **PAR_EL1**（物理地址寄存器）。

AT 指令的语法允许你指定使用哪个转换体制。例如，EL2 可以查询 EL0/EL1 转换体制。但 EL1 不能使用 AT 指令查询 EL2 转换体制（这是权限违规）。

如果请求的转换会导致错误，不会生成异常。相反，本应生成的错误类型被记录在 PAR_EL1 中。

## 知识检查

**Q：地址转换中 stage 和 level 有什么区别？**

A：stage 是将输入地址转换为输出地址的过程。Stage 1 是从 VA 到 IPA，Stage 2 是从 IPA 到 PA。Level 指的是给定转换阶段中的表层次，也是将较大块细分为较小块的方式。

**Q：物理地址的最大大小是多少？**

A：物理地址空间的最大大小是 IMPLEMENTATION DEFINED，最大可达 52 位（从 Armv8.2-A 起）。

**Q：哪个寄存器字段控制虚拟地址空间的大小？**

A：`TCR_ELx.TnSZ`，或 Stage 2 的 `VTCR_EL2.T0SZ`。

**Q：什么是转换粒度，支持哪些大小？**

A：它是可以描述的最小内存块。支持的大小为 4KB、16KB 和 64KB。

**Q：`TLBI ALLE3` 做什么？**

A：它无效化 EL3 虚拟地址空间的所有 TLB 条目。

**Q：导致 Translation Fault 的转换表条目可以缓存在 TLB 中吗？**

A：不可以。

**Q：MMU 禁用时地址如何映射？**

A：地址是平坦映射的，输入和输出地址相同。

**Q：什么是 ASID？TLB 条目何时包含 ASID？**

A：ASID 是地址空间标识符，标识转换关联的应用程序。非全局映射 (nG=1) 在 TLB 中用 ASID 标记。
